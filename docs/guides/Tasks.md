# Tasks — персональный таск-трекер

Направление `tasks` (`kind: tasks`). Личный трекер задач: две вкладки — **Текущие** (`active/`) и **Архив** (`archived/`). Одна задача = один Markdown-файл.

## Раскладка на WebDAV

`path` модуля в `config.md` — корень раздела (сейчас `Obsidian Vault/Tasks`). Внутри:

```
Tasks/
  active/            ← незакрытые задачи: Task-1.md, Task-2.md, …
  archived/          ← закрытые задачи (closed_at проставлен)
  Documents/         ← вложения, своя подпапка на задачу
    Task-3/contract.pdf
    Task-3/photo.jpg
```

- **Имя файла = id:** `Task-{N}.md`, `N` целое без паддинга. id в имени не дублируется в frontmatter.
- **Следующий id** считается сканом: `max(номеров в active/ + archived/) + 1` (без отдельного счётчика).
- **Закрытие задачи** = статус `done` + перенос файла `active/Task-N.md` → `archived/Task-N.md` (реализовано как write+delete, не WebDAV MOVE). Вложения остаются в `Documents/Task-N/`.
- **Возврат из архива** = обратный перенос, статус → `todo`.
- **Удаление** убирает только `.md`; файлы в `Documents/Task-N/` намеренно остаются.

## Формат файла задачи

Frontmatter + тело. Время — наивное, **без таймзоны, по МСК**, формат `YYYY-MM-DDTHH:MM:SS`.

Общие поля frontmatter:
- `title` — заголовок;
- `type` — `simple` | `checklist` | `steps`;
- `status` — текущий: `todo` | `in_progress` | `done`;
- `opened_at` — создание; `closed_at` — закрытие (пусто, пока активна);
- `history` — список переходов статуса `- {status, at}` (источник для фильтра «по дате смены статуса»).

Неизвестные ключи frontmatter (например будущие `priority`, `due`, `tags`, `links`) **сохраняются как есть** — парсер толерантный, ручные правки в Obsidian не теряются.

### type: simple
Тело = свободное описание; вложения — секцией `## Файлы`.
```markdown
---
title: Позвонить в банк
type: simple
status: todo
opened_at: 2026-06-08T10:30:00
closed_at: ""
history:
  - status: todo
    at: 2026-06-08T10:30:00
---

Уточнить условия по вкладу.

## Файлы
- [выписка.pdf](../Documents/Task-7/выписка.pdf)
```

### type: checklist / steps
Тело = (необязательное **описание** свободным текстом) + Markdown-список `- [ ]` / `- [x]`. Описание — это текст **до первого пункта списка** (в модели — поле `intro`); редактируется в приложении полем «Описание» так же, как у `simple`. У `steps` дополнительно: перестановка пунктов, заметка под шагом (`> …`) и вложения шага (вложенные ссылки `../Documents/Task-N/...`).
```markdown
---
title: Снять квартиру
type: steps
status: in_progress
opened_at: 2026-06-08T10:30:00
closed_at: ""
history:
  - status: todo
    at: 2026-06-08T10:30:00
  - status: in_progress
    at: 2026-06-08T11:05:00
---

Ищем 1-комнатную у метро, бюджет до 60к. ← описание (текст до списка)

- [x] Определить бюджет и район
- [ ] Собрать документы
  > нужен 2-НДФЛ
  - [contract.pdf](../Documents/Task-3/contract.pdf)
- [ ] Отсмотреть варианты
```

## Поведение в приложении

- **Список (Текущие):** кнопка «Создать задачу» (модалка: заголовок + тип, форма зависит от типа), фильтры — сортировка по дате создания / смене статуса / id (+ направление), фильтр по статусу. Карточка показывает id, тип, статус, прогресс, даты.
- **Архив:** сортировка по `closed_at` (новые сверху), пагинация (по 10, «Показать ещё»).
- **Деталь:** редактирование заголовка, переключатель статуса `todo`/`in_progress`, контент по типу, действия «Закрыть/Вернуть/Удалить», раскрывашки «история статусов» и «показать .md».
- **Сохранение** дебаунсится (~700 мс) + сейв по blur; структурные изменения (галочка, перестановка, добавление/удаление, статус, перенос) пишутся сразу. Индикатор статуса сохранения — в шапке детали.
- **Вложения** грузятся бинарно в `Documents/Task-N/` при добавлении; картинки показываются превью (тянутся через воркер), тап по имени открывает файл.

## Известные упрощения (v1, бэклог)
- Список читает все файлы `active/`+`archived/` при открытии (нет индекс-файла) — при большом архиве можно ввести индекс. Пагинация ограничивает только отрисовку.
- Толерантность тела — best-effort: сохраняются неизвестный frontmatter, intro-текст до списка и «хвост» после; произвольная глубокая разметка внутри шагов может не сохраниться побайтно.
- Поля `priority/due/tags/links` зарезервированы, но не реализованы (добавляются без миграции).
